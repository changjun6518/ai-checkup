# 지표 설계 근거 — 무엇을 어떻게 측정했고, 어디까지 믿을 수 있나

각 축이 "무엇을 재려 했고", "실제로는 무엇을 셌으며", "그 측정이 어디서 왜곡되는지"를 기록한다.
심사·리뷰 때 "이 지표 타당한가?" 질문에 대한 답변 자료.

모든 원천 데이터는 `~/.claude/projects/**/*.jsonl`의 로컬 레코드이며, 측정은 두 층으로 나뉜다:

- **기계 측정** (analyze.py/js): 레코드를 세는 것. 재현 가능, 이견 없음.
- **LLM 채점** (SKILL.md 루브릭): 샘플 프롬프트를 읽고 판단. 인사이트는 깊지만 실행마다 ±1점 변동 관측됨.

## A. 첫 프롬프트 구체성

| | |
|---|---|
| 재려는 것 | 첫 요청만으로 작업이 시작 가능한가 (되묻기 유발 여부) |
| 실제 측정 | 세션별 첫 유저 프롬프트 앞 500자를 LLM이 루브릭(목표·맥락·제약 명시 여부)으로 채점 |
| 원천 레코드 | `type: "user"` 중 content가 문자열인 첫 레코드 (툴 결과·시스템 주입은 제외) |

**타당성**: 프록시로는 합리적. 실데이터에서 기획서형 프롬프트(1,465자 구조화)와 단문 지시가 실제로 구분됐음.

**한계**:
- 첫 프롬프트만 봄 — 세션 중반의 지시 품질은 미반영.
- **길이≠구체성인데 LLM이 긴 프롬프트에 후한 경향** 있음. 루브릭 앵커("조건 명시 여부")로 완화했지만 완전하진 않음.
- 500자 절단으로 긴 프롬프트의 뒷부분 조건은 채점에서 누락.

## B. 교정 능력

| | |
|---|---|
| 재려는 것 | AI가 빗나갔을 때 요구사항을 좁혀주는 능력 |
| 실제 측정 | 프롬프트 시작부 정규식(`아니|그게 아니라|다시|틀렸…`)으로 교정 발화를 세고, 해당 프롬프트를 LLM이 채점 |
| 원천 레코드 | 유저 프롬프트 텍스트 |

**타당성**: ⚠️ **5축 중 가장 약함.** 알고 쓸 것.

**한계**:
- **오탐 확인됨**: "아니면 ~할까?"(대안 제안)가 교정으로 집계된 사례 발견. `아니면` 제외 처리 필요 (미수정).
- 미탐: "흠 그게 아니고", 말없이 새 지시로 덮어쓰기 등 다양한 교정 표현을 놓침.
- **핵심 교란**: 교정 0회가 "잘 시켜서"인지 "결과를 검수 안 해서"인지 구분 불가. 교정 0 + 높은 A점수면 전자로 추정하는 게 현재 로직이며, 리포트에 "(추정)" 표기를 강제한 이유.

## C. 모델-작업 매칭

| | |
|---|---|
| 재려는 것 | 작업 난이도에 맞는 모델 선택 (비용 감각) |
| 실제 측정 | assistant 레코드의 `message.model`로 세션별 모델 믹스 집계, 티어(fable/opus=3, sonnet=2, haiku=1) 부여. 최상위 티어 세션에서 사소한 프롬프트(인사·감사 정규식) 검출 |
| 원천 레코드 | `type: "assistant"` → `message.model`, `usage` |

**타당성**: 모델명 자체는 100% 정확한 사실 기록이라 신뢰도 높음. "Opus로 인사 12번" 같은 근거 제시가 가능한 유일한 축.

**한계**:
- trivial 정규식이 좁음(인사말 수준). "가벼운 탐색 질문에 최상위 모델" 판단은 LLM 주관에 위임됨.
- 회사 정책상 모델이 고정된 환경이면 이 축은 무의미 — 조직 배포 시 축 비활성 옵션 필요.

## D. 세션 위생

| | |
|---|---|
| 재려는 것 | 세션을 작업 단위로 관리하는 습관 |
| 실제 측정 | compaction 발생 수(`type: "system"`, subtype에 compact 포함), 세션당 프롬프트 수, 세션 제목-프롬프트 주제 일치(LLM) |
| 원천 레코드 | system 레코드, 프롬프트 카운트, `ai-title` |

**타당성**: compaction은 객관적 이벤트라 측정 자체는 견고함. "컨텍스트 50~80% 쓰면 좋다"는 초기 가설을 뒤집어 "compaction 없이 완결"을 긍정 신호로 정의한 것 — 방향은 맞다고 판단.

**한계**:
- **라이트 유저 편향**: 짧고 적게 쓰는 사람이 자동 만점. 사용량이 적은 것과 위생이 좋은 것을 구분 못 함.
- compaction이 항상 나쁜 것도 아님 — 대형 작업의 자연스러운 결과일 수 있음. 빈도가 아니라 "compaction 후에도 같은 작업을 계속 헤매는가"가 진짜 신호인데 미구현.
- 초기 아이디어였던 **컨텍스트 점유율(usage 토큰 ÷ 윈도우 크기) 자체는 미구현** — ROADMAP 참조.

## E. 고급 기능 활용

| | |
|---|---|
| 재려는 것 | 도구를 시스템으로 쓰는가 (1회성 사용자 vs 구조를 만드는 사용자) |
| 실제 측정 | plan mode 진입(`permissionMode: "plan"`), Skill/Agent tool_use 블록 수, 사용 스킬 목록 |
| 원천 레코드 | user 레코드의 permissionMode, assistant 레코드의 tool_use 블록 |

**타당성**: 카운트는 정확함. 실데이터에서도 "plan mode 0회"가 그대로 잡혀 처방으로 연결됐음.

**한계**:
- **"많이 씀 = 잘 씀" 등치가 약함.** 작은 작업만 하는 사람은 서브에이전트를 쓸 이유가 없음. 작업 규모 대비 정규화가 없어서 헤비 유저에게 유리.
- 원래 요구사항이던 "그 스킬이 업무에 도움이 됐는가"(효용)는 미구현 — 횟수만 셈.

## 공통 한계 (축 무관)

1. **표본 크기**: 세션 수가 적으면 전 축이 흔들림. 리포트에 표본 규모 명시를 강제한 이유.
2. **LLM 채점 재현성**: 동일 데이터 3회 실행에서 ±1점 변동 관측. 기계 지표 비중을 높이는 게 완화책.
3. **Goodhart 위험**: 지표를 공개하면 게이밍 가능 (예: 교정 점수 때문에 오히려 검수를 안 하게 됨). 조직 도입 시 "개인 코칭용, 평가 비연동"을 명시해야 함.
4. **원천의 강점**: 모든 수치가 로컬 jsonl의 사실 기록에서 나옴 — 자기보고식 설문과 달리 조작·미화가 불가능한 것이 이 도구의 근본 차별점.

## 한 줄 요약

자신 있게 방어 가능한 축: **C(모델 기록), D의 compaction, E의 카운트** — 전부 사실 기록.
방어에 주의가 필요한 축: **A(길이 편향), B(오탐+교란)** — "프롬프트 분석이 제일 중요"하다는 기획 의도 대비 구현이 가장 얕은 곳이기도 함. 다듬는다면 여기부터.
